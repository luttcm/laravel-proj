# Проектная документация

## 1. Архитектура системы

Система реализована как веб-приложение по клиент-серверной архитектуре.

Основные уровни:

- пользовательский интерфейс на Blade-шаблонах;
- слой маршрутизации Laravel;
- контроллеры приложения;
- сервисный слой с бизнес-логикой;
- слой репозиториев для работы с данными;
- модели Eloquent ORM;
- реляционная база данных;
- файловое хранилище для изображений и аватаров.

## 2. Основные проектные решения

### 2.1. Организация доступа

Для доступа к защищенным разделам применяются:

- стандартная аутентификация Laravel;
- middleware `auth`;
- middleware `check.access`;
- middleware `2fa`;
- middleware `role` для ограничения доступа по ролям.

Поддерживаемые роли:

- `admin`;
- `manager`;
- `finance`;
- `redactor`.

### 2.2. Подсистема расчетов

Подсистема расчетов построена на сервисном подходе.

Ключевые элементы:

- `CalculationService`;
- DTO для входных и выходных данных;
- стратегии расчета для различных типов контрагентов;
- использование справочников `Variable`, `Nds`, `Spk`.

Такое решение позволяет:

- изолировать бизнес-логику от контроллеров;
- расширять правила расчета;
- тестировать вычисления отдельно от интерфейса.

### 2.3. Подсистема контента

Контентный блок системы состоит из:

- новостей;
- комментариев;
- базы знаний;
- категорий базы знаний;
- загружаемых изображений.

База знаний поддерживает:

- иерархию "раздел - страница";
- сортировку по полю `order`;
- хранение HTML-контента;
- загрузку изображений для страницы и редактора.

### 2.4. Подсистема отчетности

Система поддерживает:

- черновики отчетов менеджеров;
- итоговые отчеты менеджеров;
- отдельные финансовые отчеты для финансового блока.

### 2.5. Подсистема справочников

Справочники служат источником нормативной информации для расчетов и отчетности:

- переменные;
- НДС;
- СПК;
- поставщики;
- компании-источники.

## 4. Модель развертывания

Для работы системы требуется:

- веб-сервер с поддержкой PHP;
- установленный PHP-интерпретатор;
- СУБД;
- директория для хранения загружаемых файлов;
- браузер на стороне клиента.

Схема взаимодействия:

1. Пользователь отправляет HTTP-запрос.
2. Маршрут Laravel направляет запрос в контроллер.
3. Контроллер обращается к сервисам, репозиториям и моделям.
4. Данные сохраняются в БД или файловом хранилище.
5. Пользователю возвращается HTML-страница или JSON-ответ.

## 5. Безопасность и контроль доступа

Решения по безопасности:

- обязательная аутентификация в защищенной зоне;
- двухфакторная аутентификация;
- ролевой контроль доступа;
- CSRF-защита форм;
- серверная валидация;
- шифрование секрета 2FA;
- скрытие конфиденциальных полей в модели пользователя.

## 6. Ограничения текущей реализации

К потенциальным ограничениям относятся:

- необходимость уточнения полного набора бизнес-правил для отчетности;
- зависимость части доступа к разделам от middleware и маршрутов;
- необходимость расширения автоматических тестов для всех ключевых модулей;
- необходимость формализации нефункциональных требований при промышленной эксплуатации.
